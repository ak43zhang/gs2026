# 新闻 AI 分析结果拆分入库与 Redis 缓存优化方案

> 版本：v2.0 | 日期：2026-04-13 | 状态：已完成

---

## 一、现状分析

### 1.1 当前架构

```
[DeepSeek AI] → 返回整批 JSON（15-18条新闻合并）
      ↓
[analysis_news2026] → 整条 JSON 作为一条记录存入 MySQL（text 字段）
      ↓
[前端新闻中心] → 从 MySQL 读取 → 需解析大 JSON → 逐条展示
```

### 1.2 当前表结构

**原始新闻表 `news_cls2026`**：
| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | text | 新闻标题 |
| 发布时间 | text | 发布时间 |
| 出处 | text | 新闻来源（原名为"来源"） |
| 内容hash | text | 内容的 MD5 哈希（无索引） |
| 内容 | text | 新闻正文 |
| analysis | text | 是否已分析（'1' 或空） |

**分析结果表 `analysis_news2026`**：
| 字段 | 类型 | 说明 |
|------|------|------|
| table_name | text | 来源表名 |
| json_value | text | **整批 JSON（15-18条混在一起）** |
| update_time | text | 更新时间 |
| version | varchar(255) | 语料版本 |

- 无主键、无索引
- 22,300 条记录，每条包含 15-18 条新闻的合并 JSON
- 实际覆盖 ~64,318 条已分析新闻

### 1.3 核心痛点

| 问题 | 影响 |
|------|------|
| **整批 JSON 存储** | 前端无法直接按条查询，需全量拉取再解析 |
| **无索引** | 全表扫描，22,300 条大 JSON 的查询极慢 |
| **字段未拆分** | 无法按评分/类型/板块筛选，不支持排序 |
| **直接读 MySQL** | 高并发下数据库压力大，响应延迟高 |
| **text 类型** | 时间字段用 text 存储，无法利用索引和时间函数 |

---

## 二、优化目标

1. **一条新闻 = 一条记录**：AI 分析后的 JSON 拆分为独立行入库
2. **字段全拆分**：评分、类型、板块等结构化字段独立列存储，支持索引和筛选
3. **Redis 缓存层**：新闻中心直接读 Redis，毫秒级响应
4. **前端逐条展示**：支持按时间、评分、类型、板块等多维度查询
5. **高级筛选**：支持评分滑块筛选、全文搜索
6. **兼容存量数据**：提供历史数据迁移方案

## 三、新表结构设计

### 3.1 拆分后的分析结果表 `analysis_news_detail_2026`

```sql
CREATE TABLE `analysis_news_detail_2026` (
  -- 主键与关联
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '自增主键',
  `content_hash` VARCHAR(64) NOT NULL COMMENT '新闻内容hash（关联news_cls2026.内容hash）',
  `source_table` VARCHAR(64) NOT NULL DEFAULT '' COMMENT '来源表名（news_cls2026/news_combine2026等）',
  
  -- 原始新闻冗余字段（避免 JOIN，加速前端读取）
  `title` VARCHAR(500) DEFAULT '' COMMENT '新闻标题',
  `content` TEXT COMMENT '新闻正文',
  `publish_time` DATETIME NOT NULL COMMENT '发布时间',
  `source` VARCHAR(64) DEFAULT '' COMMENT '新闻来源（财联社/第一财经等）',
  
  -- AI 评分字段（全部独立列）
  `importance_score` TINYINT NOT NULL DEFAULT 0 COMMENT '重要程度评分（0-15）',
  `business_impact_score` SMALLINT NOT NULL DEFAULT 0 COMMENT '业务影响维度评分（-60至60）',
  `composite_score` SMALLINT NOT NULL DEFAULT 0 COMMENT '综合评分（重要程度×4+业务影响）',
  
  -- AI 分类字段
  `news_size` ENUM('重大','大','中','小') NOT NULL DEFAULT '小' COMMENT '消息大小',
  `news_type` ENUM('利好','利空','中性') NOT NULL DEFAULT '中性' COMMENT '消息类型',
  
  -- 关联板块/概念（JSON 数组存储，支持 MySQL 5.7+ JSON 函数查询）
  `sectors` JSON COMMENT '涉及板块（JSON数组）',
  `concepts` JSON COMMENT '涉及概念（JSON数组）',
  `leading_stocks` JSON COMMENT '龙头个股代码（JSON数组）',
  
  -- 板块详情（完整的板块分析明细）
  `sector_details` JSON COMMENT '板块详情（含个股关联原因）',
  
  -- 元数据
  `analysis_version` VARCHAR(32) DEFAULT '' COMMENT 'AI分析语料版本',
  `analysis_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT 'AI分析完成时间',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '记录创建时间',
  
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_content_hash` (`content_hash`),
  KEY `idx_publish_time` (`publish_time`),
  KEY `idx_composite_score` (`composite_score`),
  KEY `idx_news_type` (`news_type`),
  KEY `idx_news_size` (`news_size`),
  KEY `idx_source_table` (`source_table`),
  KEY `idx_analysis_time` (`analysis_time`),
  -- 复合索引：前端常用查询（按时间倒序 + 类型筛选）
  KEY `idx_time_type` (`publish_time`, `news_type`),
  KEY `idx_time_score` (`publish_time`, `composite_score`),
  -- 全文索引：支持标题+内容搜索
  FULLTEXT KEY `ft_title_content` (`title`, `content`) WITH PARSER ngram
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='新闻AI分析结果明细表（一条新闻一条记录）';
```

### 3.2 设计说明

| 设计决策 | 原因 |
|---------|------|
| `content_hash` 唯一索引 | 防止同一条新闻重复入库，也用于关联原始表 |
| 冗余 `title`/`content`/`publish_time` | 前端展示时避免 JOIN 原始新闻表，单表即可完成全部查询 |
| 评分字段用数值类型 | 支持 `ORDER BY`、范围查询、聚合统计 |
| `news_size`/`news_type` 用 ENUM | 固定字典值，存储空间小，查询效率高 |
| `sectors`/`concepts` 用 JSON | 数组长度不固定，JSON 类型支持 `JSON_CONTAINS` 查询 |
| `sector_details` 用 JSON | 板块详情结构复杂，保留完整结构，前端按需解析 |
| 复合索引 `idx_time_type` | 覆盖「按时间倒序 + 按类型筛选」的最高频查询 |
| 全文索引 `ft_title_content` | 支持标题+内容的中文全文搜索（ngram 解析器） |

## 四、Redis 缓存设计

### 4.1 缓存架构

```
[AI 分析完成] → [拆分 JSON] → [写入 MySQL] + [写入 Redis]
                                                  ↓
                              [前端新闻中心] ← [读取 Redis]
                                                  ↓ (缓存未命中)
                                              [回源 MySQL]
```

### 4.2 Redis Key 设计

| Key | 类型 | 说明 | TTL |
|-----|------|------|-----|
| `news:detail:{content_hash}` | Hash | 单条新闻完整数据 | 48h |
| `news:timeline:{date}` | Sorted Set | 当日新闻时间线（score=时间戳） | 72h |
| `news:type:{date}:{type}` | Sorted Set | 按类型分组时间线（利好/利空/中性） | 72h |
| `news:top:{date}` | Sorted Set | 当日高分新闻排行（score=综合评分） | 72h |
| `news:sector:{date}:{sector}` | Set | 当日涉及某板块的新闻 hash 集合 | 72h |
| `news:latest` | List | 最新 N 条新闻的 content_hash（FIFO） | 永久 |
| `news:stats:{date}` | Hash | 当日统计（总数/利好数/利空数/重大数） | 72h |

### 5.2 筛选功能说明

| 功能 | 说明 | 默认值 |
|------|------|--------|
| **搜索框** | 全文搜索标题和内容 | 空（显示全部） |
| **评分滑块** | 拖动设置最低评分阈值（0-100） | 0（显示全部） |

### 5.3 搜索功能

- **搜索范围**：标题 + 内容全文搜索
- **搜索方式**：MySQL 全文索引（ngram 解析器）
- **支持语法**：
  - `华为` - 包含"华为"
  - `华为 芯片` - 包含"华为"或"芯片"
  - `+华为 +芯片` - 同时包含（布尔模式）
  - `华为 -美国` - 包含"华为"但不包含"美国"

## 九、性能优化

### 9.1 读取性能对比

| 场景 | 改造前 | 改造后 |
|------|--------|--------|
| 获取最新 20 条新闻 | 全表扫描 22,300 条大 JSON → **~2-5s** | Redis ZREVRANGE + Pipeline HGETALL → **~5-15ms** |
| 按类型筛选 | 全表扫描 + 应用层过滤 → **~3-8s** | Redis ZREVRANGE(type) → **~5-10ms** |
| 按评分筛选 | 不支持 | Redis ZREVRANGE(top) / MySQL 索引 → **~5-20ms** |
| 全文搜索 | 不支持 | MySQL 全文索引 → **~50-200ms** |
| 单条详情 | 不支持直接查询 | Redis HGETALL → **~1-3ms** |

### 9.2 索引优化

```sql
-- 10个索引覆盖所有查询场景
PRIMARY KEY (id)
UNIQUE KEY uk_content_hash (content_hash)
KEY idx_publish_time (publish_time)
KEY idx_composite_score (composite_score)
KEY idx_news_type (news_type)
KEY idx_news_size (news_size)
KEY idx_source_table (source_table)
KEY idx_analysis_time (analysis_time)
KEY idx_time_type (publish_time, news_type)
KEY idx_time_score (publish_time, composite_score)
FULLTEXT KEY ft_title_content (title, content) WITH PARSER ngram
```
