# 公告原文抓取工具设计方案

> 版本: V1.0  
> 日期: 2026-05-02  
> 状态: 已审核通过  
> 位置: `src/gs2026/collection/risk/notice_content_fetcher.py`

## 一、需求概述

在公告采集流程中，获取到东方财富公告网址后，调用其内容API获取公告原文全文，写入 `jhsaggg2026` 表新增字段。

## 二、技术验证

| 验证项 | 结果 |
|--------|------|
| API端点 | `https://np-cnotice-stock.eastmoney.com/api/content/ann?art_code={art_code}&client_source=web&page_index=1` |
| 响应格式 | JSON → `data.notice_content` 包含纯文本全文 |
| art_code提取 | 从`网址`字段URL末段正则提取 `(AN\d+)\.html` |
| 内容示例 | 证监会批复全文约800字 |
| PDF-only公告 | `notice_content` 为空，标记 `content_status=2` |

## 三、架构设计

### 3.1 集成位置

```
  └─ for each date:

      ├─ save2mysql(df, ...)   ← 不改
```

### 3.2 文件结构

```
src/gs2026/collection/risk/
├── notice_risk_history.py          ← 现有，改动2处
├── notice_content_fetcher.py       ← 新增
└── settings.yaml                   ← 新增（配置参数）
```

### 3.3 改动原则

- **不修改** `hsjaggg()`、`save2mysql()`、`filter_new_data()`、`safe_insert()`
- **不修改** `notice_risk_collect()` 和 `filter_gg_jhsaggg()`
- 仅在 `notice_collect()` 末尾追加3行 try-except 调用
- 内容抓取失败不影响原有采集流程

## 四、数据库变更

```sql
ALTER TABLE jhsaggg2026 
  ADD COLUMN `公告原文` LONGTEXT DEFAULT NULL COMMENT '东方财富API获取的公告全文',
```

## 七、`notice_content_fetcher.py` 核心逻辑

### 7.1 公开接口

| 函数 | 说明 |
|------|------|
| `fetch_batch_content(table_name, date, limit=2000)` | 批量抓取指定日期的公告原文 |
| `fetch_single_content(art_code)` | 单条抓取，返回 `(status, content)` |
| `backfill_content(table_name, start_date, end_date)` | 独立补跑历史数据 |

### 7.2 状态码定义

| content_status | 含义 |
|----------------|------|
| 0 | 未抓取（默认） |
| 1 | 已抓取有内容 |
| 2 | 无内容（仅PDF） |
| 3 | 抓取失败 |

### 7.3 容错设计

- API异常 → status=3，不中断批次
- 连续10次失败 → 暂停60秒后继续
- 单条重试最多3次
- 整个 `fetch_batch_content()` 被 try-except 包裹，失败只 warning 不中断主流程

### 7.4 频率控制

- 每次请求间隔 0.3-0.8 秒（随机）
- 触发暂停后等待 60 秒
- 可通过 settings.yaml 调整
