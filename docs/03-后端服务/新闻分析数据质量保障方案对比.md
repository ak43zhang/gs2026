# 新闻分析数据质量保障方案对比

> 创建时间: 2026-04-22 17:28
> 版本: 1.0
> 对比方案: A-实时拦截 vs B-离线检测

---

## 一、方案对比概览

| 维度 | 方案A: 实时拦截验证 | 方案B: 离线检测工具 |
|------|-------------------|-------------------|
| **处理时机** | 数据入库前实时验证 | 数据入库后离线扫描 |
| **拦截方式** | 异常数据不入库 | 异常数据入库后标记清理 |
| **发现时机** | 分析时立即发现 | 定时扫描或人工触发 |
| **系统影响** | 修改核心处理流程 | 独立工具，不影响主流程 |
| **灵活性** | 规则固定，需重启生效 | 规则可调，即时生效 |
| **数据完整性** | 异常数据不入库，可能遗漏 | 全量入库，事后清理 |
| **实现复杂度** | 中等（修改处理器） | 低（独立工具） |
| **维护成本** | 低（内置逻辑） | 中等（独立维护） |

## 二、方案A: 实时拦截验证

### 2.1 流程图

```
DeepSeek返回JSON
    ↓
[实时验证] ← 新增步骤
    ↓
    ├─ 验证通过 → 正常入库 → 写Redis缓存
    └─ 验证失败 → 不入库 → 记录日志 → 标记源表 → 后续重跑
```

### 2.2 验证规则

| 字段 | 理论区间 | 验证规则 | 处理方式 |
|------|----------|----------|----------|
| 重要程度评分 | 0-15 | `0 <= score <= 15` | 拦截不入库 |
| 业务影响维度评分 | -20 ~ +20 | `-20 <= score <= 20` | 拦截不入库 |
| 综合评分 | 计算值 | `composite = importance * 4 + biz_impact` | 强制重算 |
| 消息类型 | 枚举值 | `['利好', '利空', '中性']` | 拦截不入库 |
| 消息大小 | 计算值 | 根据综合评分映射 | 强制重算 |

### 2.3 修改范围

| 文件 | 修改内容 |
|------|----------|
| `news_result_processor.py` | 新增 `validate_news_record()` 函数 |
| `news_result_processor.py` | 修改 `extract_record()` 调用验证 |
| `news_result_processor.py` | 修改 `process()` 处理验证失败 |

### 2.4 优点

- 异常数据不污染数据库
- 问题发现及时，可立即重跑
- 数据质量有保障

### 2.5 缺点

- 修改核心处理流程，有风险
- 验证规则调整需重启服务
- 极端情况下可能误杀正常数据

## 三、方案B: 离线检测工具

### 3.1 流程图

```
┌─────────────────────────────────────────────────────────────┐
│                     正常分析流程                            │
│  DeepSeek返回 → 解析入库 → 写Redis缓存（不做验证）          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                   离线检测工具（独立）                       │
│  定时扫描/人工触发 → 检测异常 → 清理异常数据 → 标记源表重跑  │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 工具功能设计

#### 3.2.1 检测规则配置（JSON配置文件）

```json

      "description": "重要程度评分应在0-15之间"
    },
    {
      "name": "业务影响维度评分区间",
      "field": "business_impact_score",
      "type": "range",
      "min": -20,
      "max": 20,
      "description": "业务影响维度评分应在-20到+20之间"
    },
    {
      "name": "消息类型枚举",
      "field": "news_type",
      "type": "enum",
      "values": ["利好", "利空", "中性"],
      "description": "消息类型必须是利好/利空/中性之一"
    },
    {
      "name": "综合评分计算一致性",
      "field": "composite_score",
      "type": "formula",
      "formula": "importance_score * 4 + business_impact_score",
      "tolerance": 0,
      "description": "综合评分必须等于重要程度×4+业务影响"
    }
  ]
}
```

#### 3.2.2 工具命令行接口

```bash
# 检测指定日期数据
python news_data_validator.py --date 20260422

# 检测指定日期范围
python news_data_validator.py --start-date 20260401 --end-date 20260422

# 检测并自动清理（标记源表+删除分析表数据）
python news_data_validator.py --date 20260422 --auto-fix

# 仅检测，不清理（生成报告）
python news_data_validator.py --date 20260422 --report-only

# 交互模式（逐条确认）
python news_data_validator.py --date 20260422 --interactive
```

- 独立工具，不影响主流程

- 需要额外维护独立工具
- 发现问题有延迟

## 五、推荐方案

### 综合推荐: 方案B（离线检测工具）

**理由**:
1. 当前系统运行稳定，不宜修改核心处理流程
2. 数据质量问题非高频，离线检测足够
3. 独立工具便于迭代优化规则
4. 可逐步完善后考虑迁移为实时验证

### 实施建议

**阶段1**: 先实施方案B（离线检测工具）
- 快速落地，解决当前问题
- 积累验证规则经验
- 观察数据质量改善效果

**阶段2**: 视情况考虑方案A（实时验证）
- 当离线验证规则成熟稳定后
- 评估是否需要前置拦截
- 逐步迁移核心验证逻辑
