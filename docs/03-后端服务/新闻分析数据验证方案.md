## 一、问题背景

**案例**: "经济日报：用好我国的词元优势"
- 重要程度分理论区间: 0-15
- 实际结果: 45 (超出区间)

**根因分析**:
- DeepSeek AI 返回的评分数据可能存在异常
- 当前代码虽有 `max(-128, min(127, importance))` 等截断处理，但仅做数据截断，未标记异常数据
- 异常数据直接入库，导致后续分析偏差

---

## 二、验证规则设计

### 2.1 字段验证规则

| 字段 | 理论区间 | 验证规则 | 异常处理 |
|------|----------|----------|----------|
| `重要程度评分` | 0-15 | `0 <= score <= 15` | 标记异常，不入库 |
| `业务影响维度评分` | -20 ~ +20 | `-20 <= score <= 20` | 标记异常，不入库 |
| `综合评分` | 计算值 | `composite = importance * 4 + biz_impact` | 强制重算，不采信AI返回值 |
| `消息类型` | 枚举值 | `['利好', '利空', '中性']` | 标记异常，不入库 |
| `消息大小` | 计算值 | 根据综合评分映射 | 强制重算，不采信AI返回值 |

### 2.2 综合评分计算

```python
# 强制按公式计算，不采信AI返回的综合评分
composite = importance * 4 + biz_impact

# 消息大小映射（根据综合评分）
def _map_news_size(composite_score: int) -> str:
    if composite_score >= 50: return '重大'
    if composite_score >= 30: return '大'
    if composite_score >= 10: return '中'
    return '小'
```

## 三、验证流程

### 3.1 数据流

```
DeepSeek返回JSON
    ↓
[数据验证] ← 新增步骤
    ↓
    ├─ 验证通过 → 正常入库
    └─ 验证失败 → 标记异常 → 不入库 → 记录日志 → 后续重跑
```

### 3.2 验证函数设计

```python
def validate_news_record(msg: dict) -> tuple[bool, str, dict]:
    """
    验证新闻分析数据
    
    Args:
        msg: AI分析结果中的单条消息字典
        
    Returns:
        (is_valid: bool, error_msg: str, corrected_data: dict)
        - is_valid: 是否通过验证
        - error_msg: 错误描述（验证失败时）
        - corrected_data: 修正后的数据（验证通过时）
    """
    errors = []
    corrected = msg.copy()
    
    # 1. 验证重要程度评分
    importance = _safe_int(msg.get('重要程度评分', 0))
    if not (0 <= importance <= 15):
        errors.append(f"重要程度评分{importance}超出区间[0,15]")
    
    # 2. 验证业务影响维度评分
    biz_impact = _safe_int(msg.get('业务影响维度评分', 0))
    if not (-20 <= biz_impact <= 20):
        errors.append(f"业务影响维度评分{biz_impact}超出区间[-20,20]")
    
    # 3. 强制重算综合评分（不采信AI返回值）
    composite = importance * 4 + biz_impact
    corrected['综合评分'] = composite
    
    # 4. 强制重算消息大小
    corrected['消息大小'] = _map_news_size(composite)
    
    # 5. 验证消息类型
    news_type = msg.get('消息类型', '中性')
    if news_type not in ['利好', '利空', '中性']:
        errors.append(f"消息类型'{news_type}'不在允许值['利好','利空','中性']中")
    
    # 6. 验证必填字段
    content_hash = _safe_str(msg.get('消息id', ''))
    if not content_hash:
        errors.append("消息id为空")
    
    if errors:
        return False, "; ".join(errors), {}
    
    return True, "", corrected
```

## 四、异常数据处理

### 4.1 异常数据不入库

验证失败的数据：
- **不入库**: 不写入 `analysis_news_detail_2026` 表
- **不入缓存**: 不写入 Redis
- **记录日志**: 详细记录异常原因和原始数据
- **标记状态**: 在源表中标记为 `analysis='fail_validate'`

### 4.2 异常数据表设计（可选）

```sql
-- 异常数据记录表（用于后续重跑）
CREATE TABLE IF NOT EXISTS analysis_news_invalid_2026 (
    id INT AUTO_INCREMENT PRIMARY KEY,
    content_hash VARCHAR(64) NOT NULL,
    source_table VARCHAR(64) NOT NULL,
    raw_json TEXT NOT NULL COMMENT '原始AI返回JSON',
    error_msg VARCHAR(500) NOT NULL COMMENT '验证错误信息',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_content_hash (content_hash),
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='新闻分析验证失败记录';
```

### 4.3 源表状态标记

```python
# 验证失败时，更新源表状态
mysql_tool.update_transactions_data(
    table=source_table,
    updates={'analysis': 'fail_validate'},
    where_clause=f"content_hash='{content_hash}'"
)
```
