## 一、问题现象

涨停分析页面加载时间过长，影响用户体验。

## 二、问题排查

### 2.1 代码审查发现的问题

#### 问题1：缺少Redis缓存机制

**位置**：`ztb_analysis_service.py:get_ztb_list()`

**现状**：
- 每次请求都直接查询MySQL
- 没有使用Redis缓存列表数据
- 对于高频访问的当日数据，重复查询造成数据库压力

**影响**：
- 每次页面加载都需要执行SQL查询
- 数据量增大时响应时间线性增长

---

#### 问题2：SQL查询缺少索引

**位置**：`analysis_ztb_detail_2026` 表

**现状**：
- 查询条件：`trade_date`, `zt_time_range`, `has_expect`, `continuity`
- 仅`content_hash`有唯一索引
- 缺少复合索引支持常用查询

**当前查询**：
```sql
SELECT SQL_CALC_FOUND_ROWS 
    content_hash, stock_name, ...
FROM analysis_ztb_detail_2026
WHERE trade_date = '2026-04-13'
  AND zt_time_range = 'early'  -- 可能用到索引
ORDER BY zt_time ASC
LIMIT 20 OFFSET 0
```

**执行计划分析**：
- `trade_date` 筛选可能走全表扫描
- `ORDER BY zt_time` 需要额外排序
- `SQL_CALC_FOUND_ROWS` 增加额外开销

---

#### 问题3：JSON字段查询性能差

**位置**：`ztb_analysis_service.py:78-79`

**现状**：
```python
if sector:
    where_clauses.append(f"JSON_CONTAINS(sectors, '\"{sector}\"')")
if concept:
    where_clauses.append(f"JSON_CONTAINS(concepts, '\"{concept}\"')")
```

**问题**：
- `JSON_CONTAINS` 无法使用索引
- 需要对每条记录解析JSON
- 数据量大时性能急剧下降

---

#### 问题4：重复查询统计总数

**位置**：`ztb_analysis_service.py:96-98`

**现状**：
```python
total_sql = "SELECT FOUND_ROWS() as total"
total_df = pd.read_sql(total_sql, engine)
total = int(total_df.iloc[0]['total']) if not total_df.empty else 0
```

**问题**：
- `SQL_CALC_FOUND_ROWS` + `FOUND_ROWS()` 是MySQL的遗留功能
- MySQL 8.0 已不推荐使用
- 增加一次额外查询

---

#### 问题5：详情查询缺少缓存

**位置**：`ztb_analysis_service.py:get_ztb_detail()`

**现状**：
- 详情查询直接走MySQL
- 没有Redis缓存
- 用户反复查看同一只股票时重复查询

## 三、解决方案

### 方案A：添加Redis缓存（推荐）

#### 3.1.1 列表缓存

```python
# 缓存键设计
ztb:list:{date}:{page}:{page_size}:{filters_hash}
ztb:stats:{date}  # 统计信息缓存

# TTL设置
LIST_TTL = 5 * 60  # 5分钟，当日数据变化频繁
```

#### 3.1.2 详情缓存

```python
# 缓存键设计
ztb:detail:{content_hash}

# TTL设置
DETAIL_TTL = 30 * 24 * 3600  # 30天，历史数据不变
```

#### 3.1.3 实现代码

```python
def get_ztb_list(...):
    # 生成缓存键
    cache_key = f"ztb:list:{date}:{page}:{page_size}:{hash(filters)}"
    
    # 尝试从缓存获取
    cached = redis.get(cache_key)
    if cached:
        return json.loads(cached)
    
    # 缓存未命中，查询数据库
    result = _query_from_db(...)
    
    # 写入缓存
    redis.setex(cache_key, LIST_TTL, json.dumps(result))
    
    return result
```

**优点**：
- 大幅减少数据库查询
- 响应时间从500ms降至50ms以内
- 实现简单，改动小

**缺点**：
- 需要处理缓存一致性
- 增加Redis内存占用

---

### 方案B：数据库索引优化

#### 3.2.1 添加复合索引

```sql
-- 常用查询索引
ALTER TABLE analysis_ztb_detail_2026 
ADD INDEX idx_trade_date_zt_time (trade_date, zt_time);

-- 筛选条件索引
ALTER TABLE analysis_ztb_detail_2026 
ADD INDEX idx_trade_date_range (trade_date, zt_time_range);

-- 预期/延续性筛选索引
ALTER TABLE analysis_ztb_detail_2026 
ADD INDEX idx_trade_date_expect (trade_date, has_expect, continuity);
```

#### 3.2.2 优化统计查询

```python
# 替代 FOUND_ROWS() 的方案
def get_total_count(trade_date, filters):
    cache_key = f"ztb:count:{trade_date}:{hash(filters)}"
    cached = redis.get(cache_key)
    if cached:
        return int(cached)
    
    # 使用COUNT(*)替代FOUND_ROWS()
    count_sql = f"SELECT COUNT(*) as total FROM {table_name} WHERE ..."
    count_df = pd.read_sql(count_sql, engine)
    total = int(count_df.iloc[0]['total'])
    
    # 缓存统计结果
    redis.setex(cache_key, 60, str(total))
    return total
```

**优点**：
- 从根本上提升查询性能
- 不依赖外部缓存

**缺点**：
- 索引增加存储空间
- 写入性能略有下降
- 无法解决热点数据重复查询问题

---

### 方案C：JSON字段冗余存储

#### 3.3.1 创建冗余表

```sql
-- 板块关联表
CREATE TABLE analysis_ztb_sectors (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    content_hash VARCHAR(64) NOT NULL,
    sector VARCHAR(100) NOT NULL,
    trade_date DATE NOT NULL,
    INDEX idx_sector_date (sector, trade_date),
    INDEX idx_content_hash (content_hash)
);

-- 概念关联表
CREATE TABLE analysis_ztb_concepts (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    content_hash VARCHAR(64) NOT NULL,
    concept VARCHAR(100) NOT NULL,
    trade_date DATE NOT NULL,
    INDEX idx_concept_date (concept, trade_date),
    INDEX idx_content_hash (content_hash)
);
```

#### 3.3.2 查询优化

```python
# 原查询（慢）
WHERE JSON_CONTAINS(sectors, '"工程机械"')

# 新查询（快）
WHERE content_hash IN (
    SELECT content_hash FROM analysis_ztb_sectors 
    WHERE sector = '工程机械' AND trade_date = '2026-04-13'
)
```

**优点**：
- 彻底解决JSON查询性能问题
- 支持更复杂的关联查询

**缺点**：
- 需要维护冗余数据
- 实现复杂度高
- 改动范围大
