# P1性能优化方案 - 数据采集超时控制 + 存储异步化

## 一、问题原因分析

### 1.1 当前架构

```
主循环 (3秒周期)
│
├── 1. fetch_all_concurrently()    ← 同步阻塞，1.5-2.5s
│   ├── ThreadPoolExecutor(13线程)
│   ├── 13批 × 400只 = 5200只
│   └── as_completed() 等待所有批次
│
├── 2. 涨停判断 + 主力净额计算      ← 已优化（P0）
│
├── 3. save_dataframe()             ← 同步阻塞，0.5-1.0s
│   ├── df.to_sql() MySQL写入       ← 阻塞
│   └── redis_util.save_to_redis()  ← 阻塞
│
└── 4. culculate_gp_apqd_top30()   ← 包含额外存储
    ├── save_dataframe() × 2        ← 阻塞
    └── save_rank_to_mysql()        ← 阻塞
```

### 1.2 时间分布分析

| 阶段 | 操作 | 耗时 | 占比 |
|------|------|------|------|
| 数据采集 | fetch_all_concurrently | 1.5-2.5s | **50-60%** |
| 计算处理 | 涨停+主力净额+排行 | 0.3-0.5s | 10-15% |
| 数据存储 | save_dataframe × 3 | 0.5-1.0s | **20-30%** |
| 其他 | Redis读取+索引等 | 0.2-0.3s | 5-10% |
| **总计** | | **2.5-4.3s** | |

**核心问题**: 数据采集和数据存储占总耗时的70-90%，且都是同步阻塞。

### 1.3 风险场景

当网络波动或数据库压力大时：
- 数据采集可能超过3秒 → 周期堆积
- MySQL写入可能超过1秒 → 进一步延迟
- 最坏情况：单周期耗时5-6秒，错过下一个周期

### 方案 P1-A: 数据采集超时控制

#### 问题分析

**当前代码**:
```python
def fetch_all_concurrently(codes):
    batches = batch_codes(codes, BATCH_SIZE)  # 13批
    all_data = []
    
    with ThreadPoolExecutor(max_workers=MAX_WORKERS) as executor:
        future_to_batch = {executor.submit(fetch_batch, batch): batch for batch in batches}
        
        for future in as_completed(future_to_batch):  # 等待所有完成
            df = future.result()  # 无超时！
            if not df.empty:
                all_data.append(df)
    
    if all_data:
        return pd.concat(all_data, ignore_index=True)
    return pd.DataFrame()
```

**问题**:
1. `as_completed()` 会等待所有13批全部完成
2. 单批API超时 → 整体阻塞
3. 无总超时控制 → 最坏情况无上限
4. 每次创建新的ThreadPoolExecutor → 线程创建开销

#### 优化方案

```python
# 模块级线程池（避免重复创建）
_fetch_executor = ThreadPoolExecutor(max_workers=MAX_WORKERS)

def fetch_all_concurrently(codes, timeout=2.0):
    """
    带超时控制的并发数据采集
    
    Args:
        codes: 股票代码列表
        timeout: 总超时时间（秒），默认2秒
    """
    batches = batch_codes(codes, BATCH_SIZE)
    all_data = []
    
    futures = {_fetch_executor.submit(fetch_batch, batch): batch for batch in batches}
    
    deadline = time.time() + timeout
    
    for future in as_completed(futures, timeout=timeout):
        try:
            remaining = max(0.1, deadline - time.time())
            df = future.result(timeout=remaining)
            if not df.empty:
                all_data.append(df)
        except TimeoutError:
            logger.warning(f"单批数据采集超时")
            continue
        except Exception as e:
            logger.warning(f"单批数据采集异常: {e}")
            continue
    
    # 取消未完成的任务
    for future in futures:
        if not future.done():
            future.cancel()
    
    if all_data:
        return pd.concat(all_data, ignore_index=True)
    return pd.DataFrame()
```

#### 优势
1. **总超时2秒** — 保证不会超过周期
2. **模块级线程池** — 避免重复创建线程（节省~50ms）
3. **部分数据也能用** — 10批成功也能计算，不需要等13批全部完成
4. **异常隔离** — 单批失败不影响其他批次

#### 缺点
1. **可能丢失部分股票** — 超时时未完成的批次数据丢失
2. **数据不完整** — 某些周期只有4000只而非5000只
3. **需要处理部分数据** — 后续计算需要容忍数据缺失

#### 影响评估
- **正常情况**: 13批在2秒内完成，无影响
- **网络波动**: 10-12批在2秒内完成，丢失400-1200只股票（可接受）
- **严重超时**: 仍能获取部分数据，不会卡死

#### 预期效果
- 正常: 无变化（采集本身<2秒）
- 异常: **从无上限 → 最多2秒**（防止周期堆积）

### 方案 P1-B: 存储异步化

#### 问题分析

**当前代码**:
```python
def save_dataframe(df, table_name, time_full, expire_seconds, use_compression=False):
    # 1. 写入 MySQL（同步阻塞）
    try:
        dtype_map = {}
        for col in df.columns:       # 每次都重新计算dtype映射
            if df[col].dtype == 'object':
                max_len = df[col].astype(str).str.len().max()
                ...
        
        df.to_sql(table_name, con=engine, if_exists='append', 
                  index=False, method='multi', dtype=dtype_map)
    except Exception as e:
        logger.error(f"MySQL 写入失败: {e}")
    
    # 2. 写入 Redis（同步）
    redis_util.save_dataframe_to_redis(df, table_name, time_full, expire_seconds)
```

**问题**:
1. MySQL写入同步阻塞主线程（~300-800ms）
2. Redis写入同步阻塞主线程（~50-200ms）
3. 每次重新计算dtype映射（~30-50ms）
4. 一个周期内调用3次save_dataframe（sssj表 + top30表 + rank表）

**每周期存储总耗时**: 3次 × (300-800ms) = **0.9-2.4s**

#### 优化方案

```python
from concurrent.futures import ThreadPoolExecutor
import threading

# 存储专用线程池（2个线程：1个MySQL，1个Redis）
_storage_executor = ThreadPoolExecutor(max_workers=2, thread_name_prefix='storage')

# dtype映射缓存
_dtype_cache = {}
_dtype_cache_lock = threading.Lock()

def _get_dtype_map(df, table_name):
    """获取dtype映射（带缓存）"""
    with _dtype_cache_lock:
        if table_name in _dtype_cache:
            return _dtype_cache[table_name]
    
    dtype_map = {}
    for col in df.columns:
        if df[col].dtype == 'object':
            max_len = df[col].astype(str).str.len().max()
            varchar_len = max(10, int(max_len * 1.5)) if max_len and max_len > 0 else 30
            dtype_map[col] = sa_types.VARCHAR(varchar_len)
        elif col in ('is_zt', 'ever_zt'):
            dtype_map[col] = sa_types.SMALLINT()
        elif col == 'main_net_amount':
            dtype_map[col] = sa_types.DECIMAL(15, 2)
        elif col == 'cumulative_main_net':
            dtype_map[col] = sa_types.DECIMAL(15, 2)
        elif col == 'main_confidence':
            dtype_map[col] = sa_types.DECIMAL(3, 2)
    
    with _dtype_cache_lock:
        _dtype_cache[table_name] = dtype_map
    
    return dtype_map


def _write_mysql(df, table_name, dtype_map):
    """MySQL写入（在后台线程执行）"""
    try:
        df.to_sql(table_name, con=engine, if_exists='append', 
                  index=False, method='multi', dtype=dtype_map)
        logger.info(f"[异步] 已写入 MySQL 表 {table_name}，共 {len(df)} 条")
    except Exception as e:
        logger.error(f"[异步] MySQL 写入失败: {e}")


def _write_redis(df, table_name, time_full, expire_seconds, use_compression):
    """Redis写入（在后台线程执行）"""
    try:
        redis_util.save_dataframe_to_redis(df, table_name, time_full, expire_seconds, use_compression)
    except Exception as e:
        logger.error(f"[异步] Redis 写入失败: {e}")


def save_dataframe_async(df, table_name, time_full, expire_seconds, use_compression=False):
    """
    异步存储数据（非阻塞）
    
    MySQL和Redis写入提交到后台线程，主线程立即返回
    """
    # 获取dtype映射（带缓存）
    dtype_map = _get_dtype_map(df, table_name)
    
    # 深拷贝DataFrame（避免主线程修改影响后台写入）
    df_copy = df.copy()
    
    # 提交异步任务
    _storage_executor.submit(_write_mysql, df_copy, table_name, dtype_map)
    _storage_executor.submit(_write_redis, df_copy, table_name, time_full, expire_seconds, use_compression)
```

#### 优势
1. **主线程立即返回** — 存储在后台执行，不阻塞计算
2. **MySQL和Redis并行** — 两个写入同时进行
3. **dtype缓存** — 同一表名只计算一次映射
4. **每周期节省0.5-1.0秒**

#### 缺点
1. **数据一致性风险** — 后台写入失败时，Redis可能有数据但MySQL没有
2. **内存增加** — DataFrame深拷贝占用额外内存（~5000行 × 20列 ≈ 2MB）
3. **线程安全** — 需要确保engine是线程安全的（SQLAlchemy默认支持）
4. **写入顺序** — 不保证严格按时间顺序写入MySQL

#### 影响评估

**数据一致性**:
- Redis是主要读取源，写入快且稳定
- MySQL主要用于历史查询，延迟几百毫秒可接受
- 写入失败有日志记录，可人工补数据

**内存影响**:
- 每次拷贝约2MB，后台线程处理完后释放
- 最多同时存在2个拷贝（当前周期+上一周期未完成）
- 总额外内存: ~4MB，可忽略

**线程安全**:
- SQLAlchemy engine默认线程安全
- Redis客户端默认线程安全
- dtype_cache使用threading.Lock保护

#### 预期效果
- 每周期节省: **0.5-1.0s**
- 主线程存储耗时: 从0.5-1.0s → **<0.05s**（仅拷贝+提交耗时）

## 三、方案对比与选择

### 3.1 方案对比

| 维度 | P1-A 超时控制 | P1-B 异步存储 | P1-A+B 组合 |
|------|--------------|--------------|-------------|
| 预期节省 | 0-1.0s（异常时） | 0.5-1.0s（每次） | 0.5-2.0s |
| 实施难度 | 低 | 中 | 中 |
| 代码改动量 | ~30行 | ~80行 | ~110行 |
| 风险等级 | 低 | 中 | 中 |
| 数据影响 | 可能丢部分股票 | 写入顺序不保证 | 两者叠加 |
| 回退难度 | 低（删超时参数） | 中（需改回同步） | 中 |

### 3.2 推荐方案

**推荐: P1-A + P1-B 组合实施**

理由:
1. P1-A解决**极端情况**（网络超时），防止系统卡死
2. P1-B解决**常态瓶颈**（存储阻塞），每个周期都受益
3. 两者互不冲突，可独立回退

### 3.3 实施顺序

**Step 1**: 先实施P1-A（超时控制）— 风险低，立竿见影
**Step 2**: 再实施P1-B（异步存储）— 需要测试验证
**Step 3**: 观察运行效果，调整参数

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| MySQL写入失败 | 低 | 中 | 错误日志 + 重试机制 |
| 内存溢出 | 极低 | 高 | 限制队列大小 |
| 写入顺序错乱 | 低 | 低 | MySQL按time字段查询 |
| 程序退出时数据丢失 | 中 | 中 | 退出前等待队列清空 |

### 4.3 缓解: 优雅退出

```python
def shutdown_storage():
    """程序退出前等待存储完成"""
    logger.info("等待后台存储完成...")
    _storage_executor.shutdown(wait=True, timeout=10)
    logger.info("后台存储已完成")

# 在run_monitor_loop_synced的退出位置调用
import atexit
atexit.register(shutdown_storage)
```
